# EpubFabric 基本設計書

## 1. 文書概要

### 1.1 文書名

EpubFabric 基本設計書

### 1.2 製品名称

**EpubFabric**

読み方は「イーパブ・ファブリック」とする。

### 1.3 製品コンセプト

EpubFabricは、紙の書籍、雑誌、冊子などをスキャンして作成したPDFを解析し、見出し、本文、画像、キャプション、囲み記事などの構造を復元したEPUB 3を生成するデスクトップアプリケーションである。

名称の「Fabric」には、スキャンされた紙面の断片を識別し、文章、画像、見出しなどを再び組み合わせて、一冊の電子書籍として編み直すという意味を持たせる。

### 1.4 キャッチコピー

> スキャンPDFを、読みやすいEPUBへ編み直す。

英語表記では次を使用する。

> Scan PDF. Weave EPUB.

### 1.5 製品の位置付け

EpubFabricは、PDFをページ画像のままEPUBへ格納する単純な変換ソフトではない。

PDF内の紙面構造を解析し、文章をリフロー可能なテキストへ変換するとともに、画像、キャプション、囲み記事などの関係を復元する「電子書籍再構成ツール」と位置付ける。

---

## 2. 開発目的

スキャンPDFは、ページ全体が画像として保存されているため、そのままでは文字サイズの変更、全文検索、読み上げ、文章のコピーなどが困難である。

一般的なOCRソフトを使用すれば文字情報を取得できるが、次のような紙面構造は失われやすい。

* 章見出し、節見出し、小見出し
* 1段組み、2段組み、縦書きの読み順
* 写真、図、イラスト
* 画像に対応するキャプション
* 囲み記事
* 引用文、強調文
* 柱、ノンブル
* 脚注、注釈
* ページをまたぐ段落

EpubFabricでは、OCR、レイアウト解析、規則ベース処理、Ollamaによる意味分類、人間による校正を組み合わせ、紙面の意味構造を再構成する。

## 3. 開発方針

### 3.1 使用言語

アプリケーション本体はC#で開発する。

Pythonは実行環境に含めず、OCRおよびレイアウト解析モデルはONNX Runtimeを通じて実行する。

### 3.2 対象OS

初期バージョンはWindows 11を対象とする。

GUIにはWinUI 3を採用する。将来的なクロスプラットフォーム対応を考慮し、解析処理と画面処理は分離する。

### 3.3 ローカル処理

PDF、ページ画像、OCR結果は原則としてローカル環境内で処理する。

Ollamaもローカルで動作するものを使用し、利用者が明示的に設定しない限り、外部の生成AIサービスへ文書内容を送信しない。

### 3.4 AI依存の制限

Ollamaは補助的な意味分類に使用する。OCR、領域座標、元画像をOllamaの応答によって直接上書きしない。

AIによる分類が失敗した場合でも、規則ベース処理と手動校正によってEPUBを生成できる構成とする。

## 4. 対象文書

### 4.1 対応対象

初期バージョンでは、次の文書を対象とする。

* 書籍
* 雑誌
* 会報
* パンフレット
* 小冊子
* 報告書
* 研究資料
* 古い印刷物
* 横書き文書
* 縦書き文書
* 1段組み
* 2段組み
* 見開きスキャンPDF

### 4.2 初期対応する構成要素

* 章見出し
* 節見出し
* 小見出し
* 本文
* 画像
* キャプション
* 囲み記事
* 引用文
* 柱
* ノンブル
* 表
* 脚注
* 装飾要素

表、脚注、ルビについては、初期バージョンでは限定対応とする。

### 4.3 対象外

初期バージョンでは、次の文書を対象外とする。

* DRMで保護されたPDF
* 手書き文書
* 数式が大部分を占める専門書
* マンガの吹き出し読み順解析
* 極端に解像度が低い画像
* 大きく破損した原稿
* 完全な紙面再現を必要とする固定レイアウト出版物

```text

```

## 9. 処理工程

## 9.1 PDF読み込み

利用者がPDFを指定すると、次の情報を確認する。

* ファイル形式
* ファイルサイズ
* ページ数
* ページサイズ
* 暗号化の有無
* テキストレイヤーの有無
* ページ画像の解像度
* 見開きスキャンの可能性

テキストレイヤーが含まれている場合は、OCRを実行せず既存テキストを利用する設定も設ける。

## 9.2 ページ画像化

PDFページを300～400dpiのPNGへ変換する。

用途ごとに3種類の画像を生成する。

```text
page-original-0001.png
page-processed-0001.png
page-preview-0001.webp
```

`page-original`は画像切り出し用、`page-processed`はOCRおよびレイアウト解析用、`page-preview`はGUI表示用とする。

## 9.3 画像前処理

画像前処理は、ページごとに設定可能とする。

主な処理は次のとおりである。

* 自動回転
* 傾き補正
* ページの台形補正
* 見開き分割
* 余白除去
* 背景の均一化
* ノイズ除去
* 裏写り軽減
* コントラスト補正
* OCR用二値化

写真や図を含むページでは、元画像を保持したままOCR用画像だけを加工する。

## 9.4 レイアウト解析

ページ画像をレイアウト解析モデルへ入力し、領域候補を取得する。

初期検出分類は次のとおりである。

```text
Title
Text
Figure
Table
List
Header
Footer
Formula
```

取得した分類はそのまま最終結果とせず、EpubFabric独自の分類へ変換する。

```text
chapter_title
section_heading
subheading
body
figure
caption
aside
pullquote
table
footnote
header
footer
page_number
decorative
unknown
```

## 9.5 OCR

OCRはページ全体ではなく、検出した領域ごとに実行する。

OCR結果には次の情報を保持する。

* 認識文字列
* 行単位座標
* 文字方向
* OCR信頼度
* 使用モデル
* OCR実行日時
* 手動修正の有無

## 9.6 仮読み順

横書きでは、基本的に上から下、左から右の順に並べる。

2段組みでは、段を検出してから各段の内部を上から下へ読む。

縦書きでは、右側の列から左側の列へ進み、列内は上から下とする。

段抜き見出し、画像、囲み記事などがある場合は、規則ベースだけで確定せず、Ollamaまたは校正画面で補正する。

## 9.7 Ollamaによる意味分類

Ollamaには次の情報を渡す。

* ページ全体の縮小画像
* 対象領域の切り出し画像
* OCR文字列
* 領域座標
* レイアウトモデルの分類
* 仮読み順
* 前後領域のOCR文字列
* 横書き、縦書きの区別

Ollamaが担当する処理は次のとおりである。

* 見出しと本文の区別
* 見出し階層の判定
* 囲み記事の判定
* 画像とキャプションの関連付け
* 読み順の補正
* 柱、ノンブルの除外判定
* 画像の代替テキスト生成
* OCR結果の不自然な箇所の指摘

文章の要約、加筆、創作は行わせない。

## 9.8 文書構造化

ページ領域を、章、節、段落、画像などの文書要素へ変換する。

ページ末尾の文章が次ページへ続いている場合は、句読点、インデント、空白、文字サイズ、行間などから段落の継続を判定する。

見出しレベルに矛盾がある場合は警告を表示する。

## 9.9 校正

解析結果を利用者が確認し、修正する。

修正可能な項目は次のとおりである。

* 領域の追加
* 領域の削除
* 領域の移動
* 領域の拡大縮小
* 領域の結合
* 領域の分割
* 領域種別
* OCR文字列
* 読み順
* 見出しレベル
* 画像との関連付け
* 囲み記事の範囲
* EPUBへの収録有無

手動修正した項目は、再解析によって上書きしない。

## 11.2 スタート画面

次の操作を表示する。

* 新しいプロジェクト

* 一括変換

ページごとに次の情報を表示する。

* サムネイル

* 校正済み状態
* 除外状態

## 11.4 ページ編集画面

ページ画像上に領域枠を重ねて表示する。

領域種別は色と略称の両方で区別する。色だけに依存しない表示とする。

利用者はマウス操作で領域の追加、移動、拡大縮小、結合、分割を行える。

## 11.5 OCR編集画面

元画像とOCR文字列を並べて表示する。

OCR結果と手動修正結果は別々に保持する。

縦書き文書でも編集しやすいように、OCR文字列は横書き編集画面で表示できるようにする。

## 11.6 読み順編集

各領域上に読み順番号を表示する。

読み順はドラッグ操作、上下ボタン、番号の直接入力で変更できるようにする。

## 11.7 文書構造画面

書籍全体をツリー形式で表示する。

章や節の移動、統合、分割を可能とする。

確認できる項目は次のとおりである。

* 縦書き
* 横書き
* 文字サイズ

## 12. EPUB出力

### 12.1 出力形式

初期バージョンではリフロー型EPUB 3を生成する。

将来的に固定レイアウトEPUBにも対応する。

### 12.2 EPUB内部構成

```text
book.epub
├── mimetype
├── META-INF
│   └── container.xml
└── EPUB
    ├── package.opf
    ├── nav.xhtml
    ├── styles
    │   └── book.css
    ├── text
```

## 13. Ollama連携

### 13.1 接続先

既定の接続先は次のとおりとする。

```text
http://localhost:11434
```

### 13.2 設定項目

* Ollamaを使用するか
* 接続先
* モデル名
* タイムアウト
* 最大再試行回数
* 1回に処理するページ数
* 低信頼度ページだけに使用するか
* 画像の最大解像度
* GPU利用の有無

### 13.3 構造化出力

Ollamaの応答はJSON Schemaに限定する。

```json
{
  "pageNumber": 1,
  "writingMode": "vertical-rl",
  "blocks": [
    {
      "id": "p0001-b0003",
      "type": "section_heading",
      "headingLevel": 2,
      "readingOrder": 4,
      "relatedBlockId": null,
      "confidence": 0.93
    }
  ]
}
```

### 13.4 応答検証

次の応答は不正と判定する。

* JSONとして解析できない
* 存在しない領域IDを含む
* 許可されていない分類を含む
* 読み順が重複する
* 必須項目がない
* 信頼度が0～1の範囲外
* OCRに存在しない長文を生成している

不正な場合は再試行し、改善しない場合は規則ベース結果を使用する。

## 14. 校正対象の自動判定

次の条件を満たす領域は要確認とする。

| 条件          |    初期値 |
| ----------- | -----: |
| OCR信頼度      | 0.85未満 |
| レイアウト信頼度    | 0.80未満 |
| Ollama分類信頼度 | 0.80未満 |
| 読み順候補       |   複数存在 |
| キャプション候補    |   複数存在 |
| 見出し階層       |  前後と矛盾 |
| 領域重複        |  規定値以上 |
| 文字化け        |   検出あり |
| ページ接続       | 継続判定不能 |

しきい値は利用者が変更できるようにする。

## 16. エラー処理

| エラー           | 対応               |
| ------------- | ---------------- |

| 異常終了          | 自動保存からプロジェクトを復旧  |

## 17. 性能要件

標準対象は、A4またはB5、300dpi、200ページ程度のPDFとする。

| 項目       | 目標                |
| -------- | ----------------- |
| アプリ起動    | 5秒以内              |
| ページプレビュー | 2秒以内              |
| OCR      | GPU使用時1ページ5秒以内を目標 |
| 自動保存     | 操作後3秒程度           |
| メモリ使用量   | 通常時4GB以内を目標       |
| 復旧       | 直前の自動保存状態へ戻せること   |

処理速度はモデルとハードウェアに依存するため、進行状況、処理中ページ、残りページ数を表示する。

## 24. 設計上の基本原則

EpubFabricでは、元の情報と解析結果を分離して保持する。

次の情報を個別に保存する。

* 元画像
* 補正済み画像
* レイアウト解析結果
* OCR結果
* Ollama分類結果
* 規則ベース判定
* 利用者の修正結果
* EPUB出力結果

AIの出力によって元の文字列や座標を直接変更しない。

PDFからEPUBを直接生成せず、必ず中間文書モデルを経由する。

自動処理の精度だけではなく、誤りを発見しやすいこと、修正しやすいこと、修正結果が再解析で失われないことを重視する。

EpubFabricは、完全自動変換ソフトではなく、機械処理と人間の校正を組み合わせて、紙の文書を読みやすい電子書籍へ再構成する制作環境として設計する。
